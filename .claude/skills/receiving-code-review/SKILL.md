# レビュー指摘への対応（receiving code review）

レビュー指摘は「直すべき仮説」であって「確定した正解」ではない。**迎合（機械的同意）も盲従も無視もせず、技術的に検証してから対応する。**

## 手順

### 1. 理解する

各指摘が「何を問題と見なしているか」を正確に把握する。曖昧なら推測で動かず質問する。
複数の解釈が可能な指摘は、解釈を明示してから進める。

### 2. 検証する（最重要）

指摘が技術的に正しいかを**自分で確かめる**。コードを読む / ドキュメントを確認する / 実際に実行する。

- **正しい** → 直す
- **誤り / 前提が違う** → 根拠を添えて丁寧に反証する（黙って従わない）
- **トレードオフ（どちらも一理ある）** → 論点を明示し、判断を仰ぐか推奨を述べる

### 3. 対応する

- 迎合しない: 「おっしゃる通りです」と機械的に同意して的外れな変更をしない
- 盲従しない: レビュアー（人間・LLM 問わず）が間違うこともある。反証できるなら指摘する
- 範囲を守る: 指摘の対象だけ直す。ついでのリファクタを混ぜない

### 4. 報告する

各指摘に対して **「対応した / 対応しない（理由）/ 議論したい」** を明示する。
修正後は完了主張の前に検証コマンドで裏取りする（`~/.claude/rules/verification-before-completion.md`）。

## 禁止

- 検証せずに全指摘へ一括で「修正しました」と返す
- 自分の変更の正しさより「レビュアーを満足させること」を優先する
- 反論すべき誤った指摘に黙って従い、コードを劣化させる
