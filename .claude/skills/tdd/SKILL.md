# TDD（t-wada方式・共通コア）

言語非依存の t-wada 方式 TDD 方法論。受け入れ条件・振る舞い仕様からテストリストを導出し、Red → Green → Refactor サイクルで実装する。

> **プロジェクト固有の詳細はここには書かない。**
> テストフレームワーク・ファイル配置・テスト対象の判定基準・受け入れ条件の入力ソースは、
> 各リポジトリの `.claude/skills/tdd`（あれば）または CLAUDE.md / docs を参照する。
> このスキルは「どの言語でも変わらない方法論の骨格」だけを定義する。

## プロセス

1. **テストリスト作成** — 仕様（受け入れ条件・振る舞い）から導出
2. **Red** — 最小のテストを1つ書き、失敗を確認する
3. **Green** — 仮実装（Fake It）で最速で通す
4. **Refactor** — 三角測量で一般化し、テストを通したまま整理する

## テストリスト導出ワークフロー

### 1. 受け入れ条件の読み込み

仕様の入力ソースはプロジェクトにより異なる（GitHub Issue / 仕様書 YAML / ADR / 設計文書 等）。
**入力ソースの優先順位は各プロジェクトの規約に従う。** 仕様から以下を抽出する:

| 仕様の要素 | テストでの用途 |
|-----------|---------------|
| 背景 / 前提条件 | テストセットアップ |
| 受け入れ条件（チェックリスト） | 正常系テスト |
| エッジケース / 異常系 | 異常系テスト |
| ゴール / 期待動作 | アサーション |

### 2. テスト対象の特定

仕様が記述する振る舞いを、テスト可能な最小単位（関数 / メソッド / hook / store 等）に分解する。
「テスト対象 = 1単位」「受け入れ条件・エッジケース = 同じ単位への異なる入力・条件・検証」と捉える。

### 3. テストケースの導出

1つの対象に対して複数のテストケースを列挙する:

- 正常系（happy path）
- 異常系（バリデーション失敗・権限不足・存在しないリソース等）
- 境界値（空・最大長・null・ゼロ件等）
- 条件分岐ごとの挙動差（ロール別など）

### 4. テストリストの出力

テストリストをチェックリストとして残す（保存先・ファイル名はプロジェクト規約に従う。慣例は `_testlist.md`）:

```markdown
# <対象名> テストリスト

仕様ソース: <Issue / YAML / 文書のパス>

## <テスト対象名>

- [ ] 正常系: 説明（受け入れ条件 X 由来）
- [ ] 異常系: 存在しない ID で NotFound（エッジケース由来）
- [ ] 境界値: 空入力で空レスポンス
```

実装完了後は `- [ ]` → `- [x]` に更新する。

## 命名規約（共通方針）

- テスト名には**仕様 ID（FR/UC/SCR 等）と振る舞いの説明**を含め、仕様とトレース可能にする
- テーブル駆動 / パラメタライズdrドテストが使える言語では、ケース名に仕様 ID を入れる
- 1テスト対象 = 1テストファイルを基本とする（詳細はプロジェクト規約）

## 仕様変更時のフロー

1. 仕様（Issue / YAML / 文書）を更新
2. テストリストを差分から更新
3. Red — 新しい/変更したテストの失敗を確認
4. Green — 実装変更
5. Refactor
6. テストリストのチェックボックスを `[x]` に更新

## 禁止事項

- テストリストを作らずに実装を始めない
- Red を確認せずに Green に進まない
- Green と Refactor を同時にやらない
