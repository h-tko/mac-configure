# 本番コードへのテスト用コード混入禁止ルール（言語非依存）

## 原則

**テストのために本番コードの構造を歪めない。** テスト容易性は「環境分岐を本番に埋め込む」ことではなく「依存を抽象化して差し替え可能にする（DI）」ことで得る。

## 禁止パターン（言語共通）

### 1. 本番コード内の mock / 実装の条件分岐

実行時に「テスト用実装」と「本番実装」を `if` で切り替えるコードを本番経路に残さない。

```
// ✗ 禁止: 本番コードにテスト用分岐
func DoSomething() {
    if useMock { return doSomethingMock() }
    return doSomethingReal()
}
```

### 2. `useMock()` / `useDB()` 系の環境判定ヘルパー

`pool != nil` / `store != nil` / 環境変数などで本番 or テストを判定するヘルパーを業務オブジェクトに持たせない。

### 3. 同一機能の二重実装（`*Mock` / `*Real` サフィックス）

同じ振る舞いの mock 版と本番版を**両方**本番コードに持たない。テスト差し替えは interface 経由で行う。

### 4. 具象依存の直接保持

差し替え不能な具象（DB プール、外部 SDK クライアント等）を業務オブジェクトに直接埋め込まない。interface / port を介して保持する。

## 正しいパターン: 依存性注入（DI）

1. ドメイン固有の依存 interface（port）を定義する
2. 業務オブジェクトは interface に依存する
3. 本番用コンストラクタで具象を注入する
4. テスト用コンストラクタで mock / fake を注入する

これにより本番コードには分岐が一切残らず、テストは注入だけで成立する。

## 判断基準

迷ったら自問する:

- **「このコードを mock 無しでそのままデプロイできるか？」** → No なら設計を見直す
- **「テストのためだけに本番コードの構造を変えていないか？」** → Yes なら interface 抽出を検討
- **「同じ機能の実装が2つあるか？」** → Yes なら片方を削除し interface で抽象化
