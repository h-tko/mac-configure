# E2E テストは必ず実行し、グリーンにする（e2e must pass）

## 原則

**変更を「完了」と見なす前、および PR をマージする前に、影響範囲の E2E（エンドツーエンド）テストを実行し、グリーン（全通過）であることを確認する。** E2E が落ちたまま完了報告・マージをしない。

これはテストピラミッドの頂点での `~/.claude/rules/verification-before-completion.md`（証拠ベース報告）の適用。

## 手順

1. 変更が影響するユーザーフロー / クリティカルパスに対応する E2E を特定する
2. E2E を実行する（必要なサービス / DB / ブラウザ等を起動した上で）
3. **グリーンを確認し、結果（pass 件数）を完了報告に添える**
4. 落ちた場合の対処:
   - 実装のバグなら **実装を直す**
   - 仕様変更でテストの期待値が古いなら **テストを正しく更新する**
   - 「緑に見せるため」の skip / コメントアウト / アサーション削除・緩和は禁止

## E2E が無い / 該当しない場合の縮退

- 対象リポジトリに E2E 基盤が無い、または変更が E2E の対象外（ドキュメント・内部リファクタ等）の場合は、**利用可能な最上位のテスト（統合 → ユニットの順）で代替し、その旨を明示する**
- E2E が未整備のプロジェクトでは、整備されるまで本ルールは「最上位テストを通す」に縮退する（プロジェクト規約を優先）
- 「縮退した」こと自体を黙らない。何で代替したかを報告に書く

## 禁止

- E2E を実行せずに「動く / 完了」と報告する
- 落ちている E2E を放置してマージ / レビュー依頼に進む
- 通すためにテストを骨抜きにする（正当な理由なき skip / xfail / 削除 / アサーション緩和）

## 判断基準

- 「この変更はユーザーから見た振る舞いに影響するか？」 → Yes なら E2E 必須
- 「緑になったのは直したからか、隠したからか？」 → 隠したなら不可
- 「E2E を回さずに完了と言おうとしていないか？」 → Yes なら止まって実行する

## 補足

ユニット / 統合の作法は TDD（`~/.claude/skills/tdd`）に従う。プロジェクト固有の E2E 実行手順・コマンド（起動方法・テストランナー）は各リポジトリの CLAUDE.md / justfile を参照。
