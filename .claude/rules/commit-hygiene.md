# コミット衛生（commit hygiene）

## 原則

**1コミット = 1論理変更（atomic）。** 後から履歴を読んで「何を・なぜ変えたか」が1コミット単位で追えるようにする。無関係な変更を同じコミットに混ぜない。

## ルール

### 1. atomic commit（1コミット1論理変更）

- 1つのコミットは、単一の意味のある変更にとどめる
- 「機能追加」と「整形」「リネーム」「無関係なバグ修正」を**同一コミットに混ぜない**
- レビューしやすく、revert しやすく、cherry-pick しやすい粒度を保つ

### 2. conventional commits 形式

メッセージは `type(scope): subject` 形式を基本とする。

- 主な type: `feat`（機能）/ `fix`（バグ修正）/ `chore`（雑務）/ `refactor` / `test` / `docs` / `perf` / `style` / `ci` / `build`
- subject は命令形・現在形で「何をするか」を簡潔に（例: `fix(auth): リフレッシュトークンの失効判定を修正`）
- body には **なぜ** を書く（what はコードを見れば分かる。why が価値）
- プロジェクト規約に独自のコミット規約がある場合はそれを優先する

### 3. 無関係な変更を混ぜない

- 依頼にトレースできない変更（隣接コードの「改善」・無関係な整形・dead code 削除）を混ぜない（`~/.claude/rules/self-review-before-merge.md` / 外科的変更と連動）
- たまたま見つけた別問題は、別コミット / 別 PR / 別 Issue に分ける

### 4. squash 前提に頼らない

- 「あとで squash するから」と雑な WIP コミットを積み上げない
- 各コミットは単独でビルド / テストが通る状態を理想とする（少なくとも論理的に完結している）

## 禁止

- 1コミットに複数の無関係な変更を詰め込む
- `wip` / `fix` / `update` だけの意味のないメッセージ（何を・なぜが不明）
- 自動生成物・整形差分を機能変更コミットに混ぜる（生成物は別コミットに分離）
- シークレット・巨大バイナリ・一時ファイルを誤ってコミットする

## 判断基準

- 「このコミットメッセージを1行で正直に書けるか？」 → "and" が複数なら分割する
- 「この diff の全行が、メッセージの主題に直接トレースできるか？」 → No なら混入を取り除く
- 「半年後にこの履歴を読んで why が分かるか？」 → No なら body を足す
